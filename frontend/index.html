The Rules of the Cart

The Mathematics Behind the Bundles

How our Market Basket Analysis extracts high-converting association rules from transaction databases.

Support

Support(A) = Freq(A) / N

Measures the baseline popularity of a product in your catalog. Sets a default filter threshold of 0.05 (5%) to remove rare data noise.

Confidence

Confidence(A→B) = Supp(A∪B) / Supp(A)

Measures the conditional probability of purchase. If a user buys product A, they have a ≥ 60% probability of adding product B.

Lift

Lift(A→B) = Conf(A→B) / Supp(B)

Measures association strength. A lift ≥ 1.2 indicates that purchasing item A significantly boosts item B's purchase rate over random choice.

The Apriori Processing Engine

Our backend reads transaction sets and employs candidate itemset trimming. By trimming low-support items early (downward closure property), the engine eliminates exponential calculation overhead, scaling to massive transaction databases.

JSON REST API

A simple, CORS-enabled POST endpoint accepting cart items and returning ranked recommendations with lift and support scores in sub-milliseconds.

Zero-Latency Shimmers

The client UI displays instant shimmer loaders and smooth progress bar triggers, preventing cumulative layout shifts (CLS) on slow networks.

Pre-Calculated Rule Pickling

To avoid resource spikes, rules are calculated once and exported to model.pkl. The server loads this file directly on startup, achieving O(1) time complexity per checkout query without retraining overhead.

Frequently Asked Questions

Clear technical answers detailing our Apriori model capabilities and implementation.

Apriori works by scanning all transaction baskets to find frequent single items (support ≥ 5%), then iteratively builds pairs and triplets. It operates on the downward-closure principle: if an itemset is frequent, all of its subsets must also be frequent. Rules are filtered by lift (≥ 1.2) to ensure a strong co-purchase relationship exists.

Training a machine learning model on large datasets can cause CPU spikes and add seconds of delay. By saving the generated rules offline to a serialized pickle file (model.pkl), our Flask backend can execute query lookups in O(1) time. This ensures sub-millisecond response rates during user checkouts.
